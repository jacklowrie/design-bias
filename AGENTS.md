# Agent Guidelines

## Environment tips

- `pyproject.toml` is the entry point for all configs. Start here to learn about
  the project environment.
- The project is managed with `uv`, documented at
  [docs.astral.sh/uv](https://docs.astral.sh/uv/)
- Documentation linting rules are enforced almost everywhere. See docblocks for
  intent/use of local code.
- The repo uses `prek` for pre-commit hooks, documented at
  [prek.j178.dev](https://prek.j178.dev/usage/). You can run all code checks on
  the codebase with `uv run prek run --all-files`
- You can run the full set of checks with
  `uv run ruff check --fix && uv run ruff format && uv run ty check`.

## Core rules

- Never run python directly at the command line. Always use `uv` to run python
  code in the project env. For example, instead of
  `python3 -c "print('Hello, world!')"`, you would use
  `uv run python -c "print('Hello, world!')"`.
- For major changes, ask for approval before making a change to existing code.
- code must always pass the linter/formatter and type checker, without modifying
  rules or tool configs.
- Respect the project structure as much as possible.
- Orchestration/runnables should always go in `scripts`.
- Script outputs should always be saved to `outputs`.
- inputs to scripts should either come from `data` or `outputs`.
- reusable code should always go in a `src` module. Notebooks and scripts should
  wrap that functionality as appropriate. Please be discerning about code that
  can be reused, and prefer to put code there over locations like `scripts`or
  `notebooks` if the code could be useful in other places.
- `notebooks` are for analyzing the contents of `data` and `outputs`. Artifacts
  generated by these notebooks (i.e. plots) can be saved in `outputs`.
